Version 11

Version 11 was our 11th sprint of testing on the prototype.

The table below lists the changes between Version 10 and 11 following user feedback.

Summary of changes in v11

Reason/source
Abstraction licences page: added tabs to manage licence access A licence holder needs to manage: who can view their licences, and which licences are linked to their account. These options are added as tabs on the Abstraction Licences page.
Add or remove licence viewers New tab where a licence holder can add or revoke access to other people to view their licences. Users are added via a button. The list of users is sortable and contains remove links.
Added 'View more licences' This page caters for users who have either requested access to view licences and wish to confirm them, or wish to request more. They do this by entering a confirmation code or entering the new licence numbers.
Password reset no longer requires sign-in Simplified the user flow by removing the need for a user to sign back in after changing or resetting their password.
Registration - Updated process for organisations Based on feedback from usability testing, the process for organisations to register to view licences was reorganised. Also, new content has been incorporated based on a strong need for users to know why we request the information we do. The rest of the table covers the details.
Org Registration - Create account The account creation process has been moved to the start of the flow. This is designed to explain to the user more clearly why we need to identify them and validate their email. The pattern is copied from the 'reset password' flow.
Org Registration - Select licences Instead of starting with a single licence, an organisation user now has the ability to upload multiple licences in a single text field. This field is flexible in terms of how the licences are input.
Org Registration - Confirm licences The content on the "Are these the licences you want to view" page has been updated to make the wording more flexible and demonstrate that more licences can be added later.
Org Registration - Send security code The process of sending a security code proved very confusing in user testing. This has been simplified to a single option (postal validation), with updated content to explain why this is necessary. The user receives confirmation after the letter is sent. The Contact Preferences step has been moved out of this part of the flow to avoid confusion.
Org Registration - Enter security code Previously, the 'complete your account' and 'enter confirmation code' parts of the flow were together. This caused a lot of confusion in user testing because a password and confirmation code were seen as similar things. Instead, the 'create account' flow has been moved to the start. Also, the confirmation code field has been repositioned so that a user can now view their pending licences (without full information) before entering their code.



Version 10

Version 10 was our tenth sprint of testing on the prototype.

The table below lists the changes between Version 9 and 10 following user feedback.

Summary of changes in v10

Reason/source
Added a 'changed password' confirmation screen Previously, users received no feedback that their email address had been successfully changed.
Multiple licences view for colleagues and agents, including email and search Enables both internal colleagues to view all licences, and agents to view all licences they've been granted access to. Both types of user can filter by the email address of the licence holder, or anyone who has been granted access to view licences. They can also search for any text within the licence number or the licence name allocated by the user.
Multiple licences view for organisations and sole traders Enables organisations to view all of their licences. This works the same way as above but without the email filter.
Feedback page as iFrame Allows the Beta feedback survey to appear consistent with GOV.UK style, using an iFrame to embed it into the page.
Registration - Guidance When an individual registers an account to view their licence, they need to be informed of what the service allows them to do.
Registration - Licence number or postcode Allows an individual to use a registered licence number or postcode, to start the process of identifying their full set of licences.
Registration - No matching licence number If the service is unable to match their licences then they will need to contact us to confirm their licence information.
Registration - Select your licences Enables a user to see all licences with similar information, and confirm which of them are theirs.
Registration - Give us your email Allows a user to specify which email address to use as their username.
Registration - Contact preference Allows a user to specify which method they'd prefer us to contact them about in future (email, phone or text).
Registration - Choose security route As a user needs to confirm their identity with us, they are given the option of choosing whether to do this over the phone (followed by an email with a code) or by post, with a letter containing a confirmation code.
Registration - Request by phone If a user has chosen to verify themselves by phone, this page gives them our contact details, or the option to choose postal verification instead.
Registration - Request by post If a user chooses to receive a letter containing a code, they are told when the letter is likely to arrive, or given the choice to verify themselves by phone and email instead.
Registration - Complete account Once a user has received their 5 digit confirmation code, they need to enter it to confirm their identity.
Registration - Create password Once a user has confirmed their identity, they need to create a password to use for their online account.

Version 6

Version 6 was our sixth version prototype.

The table below lists the changes between Version 5 and 6 following user feedback.

Summary of changes in v6

Reason/source

Flow condition on summary page updated to be a summary for ‘Abstraction conditions’. A generic terms for the five complex conditions we’ve now included on the licence.

The summary splits to two paragraphs:

  • A summary of conditions that will affect the licences maximum quantities
  • A summary of environmental protection conditions that will affect the way the abstraction must be carried out

    • Below these two short paragraphs is a link to all abstraction conditions within the terms of use page.

    Licences have an array of complex conditions. For version 6 prototype we’ve decided to display and test the top five complex conditions.

    Flow condition was a generic, plain english term I applied to a cessation condition. I’ve now reverted to a non plain english option, but sticking with the names as per existing licences.

    We have a view to update to plain english, but are awaiting a content designer to manage this process.

    Help text for flow condition removed. The summary of conditions is descriptive enough to give an overview of conditions. If user testing reports confusion without this text we can look to adding it at the abstraction conditions level.

    The summary page is becoming quite full as we add in more licence data. With reference now to multiple conditions, help text against each is not feasible at this page.

    Maximum quantities before conditions has been updated to reflect maximum quantities for a licence with a time limiting condition; a condition that splits out the time limiting periods for differing abstraction amounts.

    Two show and hide links, one for each time limited period. This list would increase or decrease depending on how many times limiting conditions against the licence.

    Some licences have one set of amounts throughout, they would see a list of amounts as per version 5, without a show and hide.

    Many licences have time limiting conditions, so just listing one set of abstraction amounts won’t help licence holders know what they will be able to abstract in the future, or were abstracting under their previous time limited period.

    Terms of use page now updated with four additional complex conditions.

  • Augmentation and compensation conditions
  • Time limiting - abstraction start
  • Time limiting - conditions
  • Eel screening condition
  • Fish passage condition

    • It’s possible as we bring in more conditions we may need to rethink the design of this section within the accordion.

    Licence one and two is missing augmentation and compensation conditions, as an example of what this section would look like in a licence without such conditions. Any note towards the condition would be absent to keep the list of conditions as short as possible.

    Licences one and two have no cessation condition, but do have a note regarding no such condition, this is to raise awareness of the condition ahead of abstraction reform.

    So we can cater for displaying more licenses, most licenses have one or more of these conditions.

    Added grey lines above and below each row of data on the summary page.

    So the information scans better for the user. Quite a lot of information on the summary page now, the lines help seperate the data. They are also used across GDS services within the data patterns available for use.

    Updated cntact details page to seperate data with grey lines as per GDS data style used on the summary page.

    To remain consistant with the summary page and GDS data pattern.
